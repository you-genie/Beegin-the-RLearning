# Beegin the RLearning — 설계 문서

- **작성일**: 2026-06-15
- **상태**: 설계 확정 대기 (사용자 리뷰 단계)
- **레포명**: `beegin-the-rlearning`

---

## 1. 한 줄 컨셉

> 진짜 Python으로 강화학습(RL)을 **직접 설계**하고, 내가 짠 코드가 진짜 에이전트(꿀벌)를 학습시키는 걸 양봉장 월드에서 지켜보는 코딩 퍼즐 게임. 레벨이 오를수록 **RL의 발전사**를 따라 설계 권한과 월드 복잡도가 함께 늘어난다.

레퍼런스: *The Farmer Was Replaced* (코드로 자동화 → 점점 복잡해짐)의 RL 버전.

---

## 2. 핵심 설계 원칙

1. **"구경"이 아니라 "설계".** 기존 `CuteRL`(가시화 전용)과의 결정적 차이. 플레이어가 진짜 코드를 쓴다.
2. **진짜 코드가 진짜 실행된다.** Pyodide로 브라우저 안에서 실제 Python을 돌려 에이전트를 학습시킨다. DSL 흉내가 아님.
3. **학습이 눈에 보인다.** Q값·정책·에이전트 행동이 양봉장 그리드에 애니메이션으로 나타난다.
4. **클리어 = 정량 기준.** "학습된 에이전트의 성공률 ≥ 레벨별 임계치". 주관적 판정 없음.
5. **RL 역사 = 레벨 사다리.** 밴딧 → TD/Q-learning → 셰이핑 → 표현설계 → 함수근사 → 정책기반 → 심층RL → 멀티에이전트 → 환경설계 순으로 설계 권한이 열린다. 단일 에이전트 → 멀티 에이전트 진행도 실제 RL 발전사와 일치.
6. **레벨은 데이터다.** 챕터 추가가 코드 수정이 아니라 데이터(레벨 정의) 추가로 끝나도록 설계.

---

## 3. 테마: 양봉장 (Apiary)

에이전트 마스코트 = 꿀벌. 양봉 메커니즘이 RL 개념에 자연스럽게 매핑되며, 난이도 상승 요소(말벌·과열·분봉)가 테마에 내장되어 있다.

| 양봉 메커니즘 | RL 개념 | 챕터 | 에이전트 |
|---|---|---|---|
| 🌼 일벌이 꽃밭 고르기 (꿀 양 숨겨짐) | 멀티암드 밴딧 — 탐험 vs 활용 | 1 (MVP) | 단일 |
| 🍯 벌집↔꽃 길찾기 + 꿀 배달 | Tabular Q-learning | 2 (MVP) | 단일 |
| 💃 먼 꽃밭이라 sparse → 웨글댄스 보상 셰이핑 | Reward shaping | 3 (MVP) | 단일 |
| 👑 산란 vs 채집 자원 배분 | 다목적 reward 설계 | 중반 | 단일 |
| 🔥 과열/쪄죽기 (부채질 vs 뭉치기) | 연속 제어·항상성 | 중반 | 단일 |
| 🐝⚔️ 말벌 침입 방어 (벌공) | 적대적/방어 정책 | 후반 | 멀티 |
| 🐝🐝 분봉 (새 둥지 합의, 정족수) | 멀티에이전트·창발 | 후반 | 멀티 |
| ☠️ 일벌의 여왕 교체 (supersedure) | 메타: 정책 실패 시 교체 | 끝판/메타 | 멀티 |

MVP(챕터 1~3)는 전부 **일벌 한 마리** 단위.

## 4. 핵심 게임 루프

1. 플레이어가 IDE(Monaco)에 Python 코드 작성 — 챕터 1~3은 `reward(...)` 함수.
2. **Run** 클릭 → 코드가 Pyodide에서 실제 실행되어 에이전트를 N 에피소드 학습.
3. 학습은 청크 단위로 진행하며 스냅샷(Q-table, 샘플 rollout)을 JS로 전달 → 양봉장 그리드에 **학습되는 과정**을 애니메이션으로 표시 + 학습곡선 그래프.
4. 학습 종료 후 평가 에피소드로 **성공률** 측정.
5. 성공률 ≥ 레벨 임계치 → 클리어, 다음 레벨 해금. 실패 → 힌트와 함께 재시도.

## 5. 기술 스택 & 실행 모델

- **Vite + TypeScript** — 빌드/개발 서버.
- **Pyodide** — 브라우저 안에서 진짜 Python 실행. 플레이어 코드 + RL 엔진 모두 Python.
- **Monaco Editor** — 코드 IDE (구문 강조, 자동완성).
- **Canvas 2D** — 양봉장 그리드·Q값·정책·리플레이 렌더링.
- **백엔드 없음** — 전부 클라이언트 사이드. 정적 호스팅 가능.

**선택 근거**: "진짜 Python으로 진짜 RL을 짠다"는 핵심 판타지를 가장 정직하게 살리면서 설치 0. Tabular RL은 Pyodide에서 충분히 빠르다. (대안으로 검토했던 데스크톱+로컬Python은 배포 부담, JS 엔진은 "진짜 RL" 감성 약함, Unity+DSL은 진짜 코드가 아님 → 모두 기각.)

**보안**: 플레이어 코드는 Pyodide WASM 샌드박스 안에서만 실행되어 호스트 시스템 접근 불가.

## 6. 아키텍처

```
[ Monaco 에디터 ] ──플레이어 코드──▶ [ Pyodide (Python 런타임) ]
                                          │  engine/  ← 고정된 RL 기계 (env + 학습 알고리즘)
                                          │  player_code ← 플레이어가 짠 reward()
                                          ▼
        run_training(player_code, level_config)
            → { success_rate, learning_curve[], q_snapshots[], sample_rollout, error }
                                          │
[ Canvas 렌더러 ] ◀──────결과 JSON──────┘   (그리드 + Q값 + 정책 화살표 + 리플레이)
[ 레벨/채점 시스템 ]   성공률 ≥ 임계 → 클리어 → 다음 레벨 해금
```

### 구성 요소 (독립적으로 이해·테스트 가능하게 분리)

| 모듈 | 언어 | 책임 | 의존 |
|---|---|---|---|
| `engine/` | Python | 환경(bandit, gridworld) + 학습 알고리즘 + `run_training` 엔트리포인트 | Pyodide(numpy) |
| `levels/` | TS (데이터) | 레벨 정의: env, 엔진 설정, 코드 템플릿, 성공 임계, 힌트, 시각화 종류 | 없음 |
| `runtime/` | TS | Pyodide 로드·플레이어 코드 주입·`run_training` 호출·결과 파싱 | Pyodide |
| `render/` | TS | 그리드·Q값·정책 화살표·리플레이 애니메이션·학습곡선 | Canvas |
| `ui/` | TS | 에디터·Run 버튼·채점 표시·레벨 진행도·힌트 패널 | 위 모듈들 |

### Python ↔ JS 경계

- 플레이어 코드(Python)는 엔진이 정의한 시그니처를 채운다. 챕터 1~3: `reward(...)`.
- `run_training`은 **JSON 직렬화 가능한 결과**만 반환 (TS에서 그대로 렌더).
- 학습 진행 애니메이션: 청크 단위로 스냅샷을 모아 반환 → JS가 순차 재생. (실시간 콜백 스트리밍은 MVP 이후 고려.)

## 7. 교육 레이어 — "보여주고 → 코딩 → 이름표"

**대전제: 플레이어는 RL을 전혀 모른다.** 용어("밴딧", "Q-learning")를 먼저 던지지 않는다. 양봉 상황으로 먼저 겪게 하고, 클리어한 뒤에야 이름표를 붙인다.

모든 서브레벨은 3단계 구조를 가진다:

| 단계 | 내용 |
|---|---|
| **① Show (데모)** | 코딩 전, 상황을 보여주는 짧은 관전/인터랙티브 데모. "벌이 뭘 하려는데 잘 안 되네?"를 체감시킨다. 용어 없음. |
| **② Code (코딩)** | 함수 빈칸 채우기. 서브레벨마다 새 개념 딱 하나만. 시작 코드 템플릿 + 단계적 힌트 제공. |
| **③ Name it (리캡)** | 클리어 후: "방금 한 게 RL에서 **~**예요." 용어 + 실제 RL에서의 의미/역사 한 스푼. |

힌트는 단계적으로 노출(막혔을 때만). 실패 시 무엇이 잘못됐는지 시각화로 보여준다 (예: 벌이 빙빙 돈다).

## 8. MVP 서브레벨 상세 (챕터 1~3, 총 9개)

각 챕터당 3개 서브레벨. 각각은 작고, 새 개념 하나씩만 도입.

### 챕터 1 — 일벌의 선택 (밴딧 / 상태 없는 단발 선택)
엔진(고정): ε-greedy action-value 학습. 플레이어는 `reward(...)`만.

| 서브 | 제목 | Show | Code (플레이어가 짜는 것) | Name it (리캡) |
|---|---|---|---|---|
| **1-1** | 보상이 뭐예요? | 벌이 꽃밭을 랜덤으로 고른다 — 아무 기준 없이 헤맴 | `reward(nectar)` → 얻은 꿀을 그대로 리턴 (거의 빈칸 채우기) | "보상(reward) = 에이전트가 극대화하려는 점수 신호" |
| **1-2** | 새 꽃도 가봐야지 | 한 꽃만 고집하는 벌 — 더 좋은 숨은 꽃을 못 찾음 | reward에 **안 가본 꽃 탐험 보너스** 추가 | "탐험 vs 활용 (exploration/exploitation)" |
| **1-3** | 나쁜 보상의 함정 | "꽃 많이 방문"에 보상 → 벌이 꿀은 안 모으고 꽃만 깨작 (reward hacking) | 목표(벌집에 꿀 전달)와 **정렬된** reward로 수정 | "보상 정렬 + 이런 문제를 멀티암드 밴딧이라 불러요 🎰" |

- **시각화**: 꽃밭별 추정 가치 막대, 선택 빈도, 누적 꿀.
- **클리어**: 평가 구간 최적 꽃밭 선택률 ≥ 임계(예: 80%).

### 챕터 2 — 꿀 배달 (Q-learning / 그리드)
엔진(고정): Tabular Q-learning (lr, γ, ε 고정). 플레이어는 `reward(state, action, next_state, done)`.

| 서브 | 제목 | Show | Code | Name it |
|---|---|---|---|---|
| **2-1** | 꽃까지 가는 길 | 작은 그리드, 벌이 무작위로 헤맴 | 목표(꽃) 도달 시 +1 리턴 | "Q값이 목표에서 거꾸로 퍼진다 (TD 학습)" |
| **2-2** | 집까지 배달 | 꽃만 찍고 안 돌아옴 | 꽃→벌집 **2단계** 전달에 보상 | "순차적 보상 / 단계적 목표" |
| **2-3** | 거미줄 피하기 | 위험칸(거미줄)에 걸려 죽음 | 위험칸에 **음의 보상** 추가 | "패널티 = 음의 보상, 그리고 이게 Q-learning이에요" |

- **시각화**: 칸별 maxQ 히트맵, 정책 화살표, 학습된 경로 리플레이, 에피소드 보상 곡선.
- **클리어**: 평가 에피소드 목표(배달) 달성률 ≥ 임계.

### 챕터 3 — 멀리 핀 꽃 (Reward shaping / sparse)
엔진(고정): 챕터2와 동일하되 맵이 크고 에피소드 예산 제한.

| 서브 | 제목 | Show | Code | Name it |
|---|---|---|---|---|
| **3-1** | 너무 멀어요 | 큰 맵 + +1-only → 예산 안에 한 번도 못 도달, 학습 실패 | (먼저 실패를 체험만) | "보상이 드물면(sparse) 학습이 안 된다" |
| **3-2** | 거리 힌트 주기 | — | 목표까지 **거리 기반 셰이핑** 항 추가 → 성공 | "보상 셰이핑 (reward shaping)" |
| **3-3** | 셰이핑의 함정 | 나이브한 셰이핑 → 벌이 보상만 빨며 빙빙 돔 (reward hacking) | **potential-based** 셰이핑으로 수정 | "잘못된 셰이핑의 위험 + potential-based shaping (Ng, 1999)" |

- **시각화**: 챕터2 + sparse(실패) vs shaped(성공) 학습곡선 비교.
- **클리어**: 에피소드 예산 내 목표 도달률 ≥ 임계.

## 9. 레벨 정의 데이터 형태 (개념)

각 서브레벨은 다음을 담은 데이터 객체로 정의되어, 챕터/서브레벨 추가가 데이터 추가로 끝나도록 한다.

| 필드 | 의미 |
|---|---|
| `id`, `chapter`, `sublevel`, `title` | 식별·표시 (예: `2-3`) |
| `env` | 환경 종류와 파라미터 (bandit arms / grid 레이아웃 등) |
| `engineConfig` | 알고리즘·하이퍼파라미터·에피소드 예산·무엇이 잠겨있고 무엇이 노출되는지 |
| `showDemo` | ① Show 단계: 코딩 전 보여줄 데모 시나리오/연출 |
| `codeTemplate` | ② Code 단계: 시작 코드 (채워야 할 함수 시그니처) |
| `successThreshold` | 클리어 성공률 기준 |
| `hints[]` | 단계적 힌트 (막혔을 때만 노출) |
| `recap` | ③ Name it 단계: 클리어 후 붙일 용어·개념·역사 한 스푼 |
| `visualizations[]` | 이 서브레벨에서 보여줄 시각화 종류 |

## 11. 열린 질문 / 리스크

- **Pyodide 초기 로딩 시간** — 첫 로드가 수 초 걸림. 로딩 UI 필요. (학습 속도 자체는 tabular라 문제 없음.)
- **플레이어 코드 에러 핸들링** — 문법/런타임 에러를 IDE에 친절하게 표시하는 UX 필요.
- **성공 임계치 튜닝** — 너무 빡세면 좌절, 너무 느슨하면 학습 의미 퇴색. 레벨별 플레이테스트로 조정.
- **reward hacking을 "버그"가 아니라 "교육 포인트"로** 보여주는 연출 (특히 Lv3).
